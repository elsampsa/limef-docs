Detailed Description

DecodedToTensorFrameFilter — converts DecodedFrame → TensorFrame (c, h, w)

Author
Sampsa Riikonen
Date
2025
Version
1.6.1

Design

The filter auto-detects the incoming pixel format and selects the appropriate internal converter (strategy pattern). Users interact only with the filter itself — converters are an implementation detail.

Output is always a single-plane TensorFrame with shape (c, h, w):

  • c = 1 for greyscale, c = 3 for colour
  • dtype = UInt8

Supported input formats

CPU: AV_PIX_FMT_GBRP, AV_PIX_FMT_GRAY8 (see note below) GPU: AV_PIX_FMT_CUDA / NV12 (requires LIMEF_CUDA)

Any unrecognised format causes a one-time error log and the filter deactivates (drops all subsequent DecodedFrames) until the format changes.

Non-DecodedFrame types (SignalFrame, StreamFrame, …) always pass through.

CPU path: prepend SwScaleFrameFilter

The CPU converters expect the pixel format to already be normalised to either GBRP (colour) or GRAY8 (greyscale). For any other input format — YUV420P from a software decoder, YUYV422 / MJPEG from a webcam, NV12, packed RGB, etc. — prepend a SwScaleFrameFilter to perform the format conversion first:

DumpFrameFilter dump ("dump");
DecodedToTensorFrameFilter d2t ("d2t", ChannelOrder::RGB);
SwScaleFrameFilter scale("scale", AV_PIX_FMT_GBRP);
d2t.cc(dump);
scale.cc(d2t); // chain: scale → d2t → dump

This two-step approach (SwScale normalises, converter repacks) performs two memcpy passes over the frame. That is acceptable for typical resolutions and frame rates; the H2D transfer to GPU dominates in GPU-inference pipelines anyway.

If a single-copy CPU path is ever needed (e.g. to reduce latency at very high resolutions), direct converters for YUV420P, NV12, etc. could be added here. They would need SIMD (SSE/AVX/NEON) to be competitive with SwScale's hand-tuned routines, so the added complexity is only worthwhile when profiling confirms the bottleneck.
