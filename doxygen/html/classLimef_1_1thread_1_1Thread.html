Detailed Description

template<typename T>
class Limef::thread::Thread< T >

A multithreading base class.

Thread class includes:

  • thread "frontend" methods: you are supposed to call these from the main thread.
  • thread "backend" methods: these are used internally by the thread after it has been spawned.
  • a single, unified fifo for handling incoming Frames and commands, the FrameFifo.

thread intercom

The only communication channel of the thread to the "outside" world (i.e. to main thread) is the FrameFifo.

Thread backend receives a continuous stream of Frames into its FrameFifo and based on those Frames it performs actions. These can be, for example, the decoding of the Frame and sending it downstream, or if the Frame is a SignalFrame, perform some action, for example, to stop the thread: it is important to note that thread control is also achieved by streaming frames into the FrameFifo.

templating

While a thread class reads and accepts all frames from it's FrameFifo, it is templated by a special frame type.

The special frame type is handled in another way than all the other frame types: if an incoming frame belongs to the special frame type, an internal frame of that type is pulled from the FrameFifo, and a copy of the incoming frame is made into the internal frame. After this, that internal frame is placed into a fifo for further processing and sent downstream.

Why do things in such complicated way?

For frame types such as DecodedFrame or PacketFrame, we want to avoid the constant (re)allocations of huge chunks of memory as these frames typically require lots of heap memory and can arrive at very high frames-per-second rate.
So a copy of them is made, instead of constantly (re)allocation them. DecodedFrame and PacketFrame would be typical template parameters for a decoding and a media source threads, respectively.

A SignalFrame on the other hand, arrives very seldomly, so such copying avoiding reallocation is not necessary.

A note on the implementation: the said logic of handling special frame class differently, is implemented in the Limef::FrameFifo class, and so the Limef::thread::Thread implementation looks pretty simple.

The input framefilter of the thread is connected to the framefifo in the ctor.

periodic tasks

In addition to receiving frames, processing them and sending them downstream, a thread can have an internal, periodic task, defined in Thread::task. The processing of incoming frames and the periodic execution of the task is "interleaved" and guaranteed to happen always in time. For example implementation of a periodic task, see ExampleThread and ProducerThread source codes.

thread usage example

An example how you would use a thread subclass named ExampleThread:

ExampleThread exampleThread("example", FrameFifoContext(false, 5, 10));
exampleThread.start(); // starts the thread "backend"
std::this_thread::sleep_for(std::chrono::seconds(1));
exampleThread.pingCall(); // frontend method: sends a signalframe seamlessly to the backend
std::this_thread::sleep_for(std::chrono::seconds(1));
exampleThread.pingCall2(2);
std::this_thread::sleep_for(std::chrono::seconds(10));
exampleThread.stop(); // sends a signalframe to the backend telling the thread to stop
